# ai-service 开发流程阶段

## 1. 文档目标

本文档仅面向 `ai-service` 模块，定义算法服务从环境准备到可联调、可演示的分阶段开发流程。

约束如下：

- 只修改 `ai-service` 目录内代码和与其直接相关的本地运行环境。
- 不修改 `backend`、`frontend`、`docs`、`openspec` 中的实现。
- 与后端、前端的交互仅以现有文档约定为准，当前阶段通过约定接口和本地 mock 自检。

---

## 2. 当前现状

当前 `ai-service` 已具备以下骨架：

- `app.py`：仅有 Flask 健康检查接口
- `config.py`：仅有基础服务配置
- `modules/`：已预留
  - `stream_reader.py`
  - `person_detector.py`
  - `face_recognition_service.py`
  - `helmet_detector.py`
  - `zone_detector.py`
  - `fall_detector.py`
  - `running_detector.py`
  - `backend_client.py`

当前问题：

- 所有核心模块仍是 `NotImplementedError`
- 缺少真实模型依赖
- 缺少统一推理编排流程
- 缺少本地测试样例和调试脚本
- 缺少人脸库、本地配置、结果缓存等运行配套

---

## 3. 总体目标

`ai-service` 的目标不是单个模型能跑，而是形成一条最小可演示闭环：

1. 读取视频流或本地视频
2. 检测人员并生成稳定 `trackId`
3. 完成人脸识别与陌生人判断
4. 完成头盔、区域、摔倒、跑动等风险识别
5. 输出统一 JSON 结果
6. 上报后端接口或写入本地 mock 结果
7. 支持本地调试、回放和日志排查

---

## 4. 分阶段开发流程

### 阶段 0：环境准备

目标：让 `ai-service` 具备可安装、可运行、可调试的本地环境。

建议任务：

- 补全 `requirements.txt`
- 明确 Python 版本
- 明确 CPU / GPU 运行模式
- 新增模型目录规范
- 新增测试样例目录规范
- 新增环境变量说明

建议补充的依赖方向：

- `opencv-python`
- `numpy`
- `Pillow`
- `ultralytics`
- `insightface`
- `onnxruntime` 或 `onnxruntime-gpu`
- 需要时再补追踪或姿态依赖

建议目录：

```text
ai-service/
  data/
    samples/
    faces/
    outputs/
  models/
    yolo/
    insightface/
    helmet/
  modules/
```

阶段产出：

- 环境可安装
- `python app.py` 可启动
- 基础目录结构清晰

---

### 阶段 1：视频读取与基础调试

目标：先打通“读流/读视频 -> 拿到帧”。

建议任务：

- 实现 `modules/stream_reader.py`
- 支持本地视频文件读取
- 支持 RTSP/RTMP/HLS 读取
- 支持断流重连
- 输出 `frame`、`frameId`、`timestamp`
- 加入基础日志

优先级建议：

- 先支持本地视频文件
- 再支持摄像头/流媒体地址

阶段产出：

- 能连续读取视频帧
- 断流时有明确日志
- 可以控制抽帧频率

---

### 阶段 2：人员检测主干

目标：建立整个算法服务的主干结果，因为后续人脸、头盔、区域、动作都依赖人员框。

建议任务：

- 实现 `modules/person_detector.py`
- 使用 YOLO 输出 `person` 检测框
- 生成统一 bbox 结构
- 提供 `centerPoint` 与 `footPoint`
- 引入简单追踪，保证 `trackId` 基本稳定

最低输出要求：

- `type`
- `trackId`
- `bbox`
- `centerPoint`
- `footPoint`
- `confidence`

阶段产出：

- 对单帧和视频都能输出 `PERSON_DETECTION`
- 结果结构对齐接口文档

---

### 阶段 3：人脸识别与陌生人判断

目标：完成你负责的核心身份识别能力。

建议任务：

- 实现 `modules/face_recognition_service.py`
- 支持本地加载员工人脸特征库
- 支持从图片提取人脸特征
- 支持人脸库缓存和热重载
- 根据阈值判断 `matched` 或 `unknown`
- 输出 `employeeId`、`similarity`、`matched`

建议拆分为两个子步骤：

1. 先做人脸特征提取与本地比对
2. 再做与人员框、轨迹结果绑定

阶段产出：

- 已知员工可识别
- 未匹配人员返回陌生人结果
- 可输出 `FACE_RESULT`

---

### 阶段 4：头盔检测

目标：补齐安全着装识别。

建议任务：

- 实现 `modules/helmet_detector.py`
- 明确检测方式
  - 方案 A：整个人框内识别 `helmet/no_helmet`
  - 方案 B：头部区域二次检测
- 为每个 `trackId` 输出头盔状态
- 增加连续帧稳定策略，减少闪烁误报

阶段产出：

- 可输出 `HELMET_WARNING`
- 支持阈值调整

---

### 阶段 5：区域检测

目标：补齐危险区域判断，但仍然只在 `ai-service` 内实现算法逻辑，不改后端区域管理。

建议任务：

- 实现 `modules/zone_detector.py`
- 支持从本地 JSON 读取区域配置
- 基于 `footPoint` 做点在多边形内判断
- 支持安全距离计算
- 输出 `inside`、`distance`、`safeDistance`

阶段产出：

- 可输出 `ZONE_WARNING`
- 同一路视频可加载不同区域配置

---

### 阶段 6：摔倒与异常跑动

目标：补齐动作类风险识别。

建议任务：

- 实现 `modules/fall_detector.py`
- 实现 `modules/running_detector.py`
- 保存每个 `trackId` 的连续帧历史
- 跑动基于位移速度和持续帧数判断
- 摔倒基于宽高比变化、姿态趋势或连续状态判断

建议原则：

- 先做规则版可演示实现
- 后续再替换为更强模型

阶段产出：

- 可输出 `FALL_ALERT`
- 可输出 `RUNNING_ALERT`

---

### 阶段 7：统一编排与结果上报

目标：把前面所有模块串成一条真正可运行的推理流水线。

建议任务：

- 新增统一编排模块，例如：
  - `modules/pipeline.py`
  - 或 `modules/frame_processor.py`
- 实现单帧处理顺序：
  - 读帧
  - 人员检测
  - 人脸识别
  - 头盔检测
  - 区域判断
  - 摔倒判断
  - 跑动判断
  - 汇总 JSON
- 使用 `modules/backend_client.py`
- 支持两种输出模式
  - 上报后端 API
  - 写入本地 JSON 调试文件

建议优先顺序：

1. 本地保存结果
2. 再接真实 HTTP 上报

阶段产出：

- 统一结果格式
- 单路视频可稳定连续产出结构化结果

---

### 阶段 8：本地验证与演示准备

目标：让服务进入“可自测、可演示、可排障”状态。

建议任务：

- 准备固定测试视频
- 准备最小人脸样本库
- 记录各能力的阈值
- 补充失败场景日志
- 保存输出样例
- 编写运行说明

至少准备以下验证样例：

- 单人正常出现在画面中
- 多人同时出现
- 已登记员工识别
- 陌生人识别
- 未戴头盔
- 区域入侵
- 摔倒
- 跑动

阶段产出：

- 演示视频输入稳定
- 输出 JSON 可复现
- 错误日志可定位问题

---

## 5. 推荐开发顺序

建议严格按下面顺序推进：

1. 环境与依赖
2. 视频读取
3. 人员检测与 `trackId`
4. 人脸识别与陌生人判断
5. 结果保存到本地 JSON
6. 头盔检测
7. 区域检测
8. 摔倒检测
9. 异常跑动检测
10. HTTP 上报

原因：

- 人员检测是所有下游能力的基础
- 人脸识别是你当前负责的主线能力
- 先本地出结果，再接远端接口，排障更简单

---

## 6. 每阶段完成标准

### 最低完成标准

- 每一阶段都能单独运行
- 每一阶段都有可观察输出
- 每一阶段失败时有明确日志

### 可联调标准

- 结果字段名固定
- 时间戳格式统一
- `trackId` 在连续帧中基本稳定
- 人脸识别结果能和人员轨迹绑定
- 每条风险结果都能映射到具体 `cameraId` 和 `trackId`

### 可演示标准

- 输入固定视频时结果稳定
- 陌生人、头盔、区域、摔倒、跑动至少各有一个可复现案例
- 服务中断或模型加载失败时不会无声崩溃

---

## 7. 开发注意事项

- 不要一开始就同时接所有模型，先保证主干链路可跑
- 不要把后端逻辑写进 `ai-service`
- 不要直接依赖数据库
- 区域判断必须优先使用 `footPoint`
- 跑动和摔倒必须基于连续帧历史，不能只看单帧
- 陌生人判断要保留阈值配置，不要把阈值写死
- 先解决“能稳定输出”，再追求“检测更强”

---

## 8. 下一步建议

如果按当前任务继续推进，`ai-service` 建议优先落这 5 件事：

1. 补 `requirements.txt`
2. 实现 `stream_reader.py`
3. 实现 `person_detector.py`
4. 实现 `face_recognition_service.py`
5. 新增统一编排模块并支持本地 JSON 输出

这样可以最快把你负责的人脸主线先跑通，再逐步补头盔和动作能力。
